# 二期功能讨论

本文记录二期候选功能的可行性讨论，供后续排期参考。当前一期（MVP）以本地 Markdown、GitHub 手动同步、TTS 朗读为主，见 [CHANGELOG.md](../CHANGELOG.md)。

---

## 1. Notion 式「笔记内容拖拽」

### 需求澄清

此处指 **编辑器内** 拖动段落/块调整顺序（Notion 左侧 ⋮⋮ 拖块），**不是** 列表里把笔记拖进文件夹。

两种能力成本差异很大：列表拖文件夹属于中等成本；**正文块拖拽属于高成本**，接近重写编辑器。

### 当前编辑器架构

[`EditorScreen.kt`](../app/src/main/java/com/andriod/reader/ui/editor/EditorScreen.kt) 正文为 **单个** `BasicTextField`：

- 数据模型：一整段 Markdown 字符串（`TextFieldValue`）
- 格式工具栏：在光标处插入 `**`、`- ` 等 Markdown 符号
- 撤销/重做：整篇文本快照（[`EditorUndoHistory.kt`](../app/src/main/java/com/andriod/reader/ui/editor/EditorUndoHistory.kt)）
- 持久化：[`MarkdownParser`](../app/src/main/java/com/andriod/reader/data/local/MarkdownParser.kt) 读写 `.md` 文件

**没有「块（Block）」抽象**，系统无法识别哪一段是可独立拖动的单元，因此无法在现有模型上「只加一个拖动手势」实现 Notion 式体验。

Notion 的做法是：段落/标题/列表项等各是一个 block → UI 层可 reorder → 再序列化存盘。

### 若要实现，至少需要

```mermaid
flowchart LR
    md[Markdown 字符串]
    blocks[Block 列表模型]
    ui[每块可编辑 UI + 拖动手柄]
    drag[拖拽重排]
    md -->|打开 parse| blocks
    blocks --> ui
    ui --> drag
    drag --> blocks
    blocks -->|保存 serialize| md
```

| 模块 | 工作 | 主要难点 |
|------|------|----------|
| 块解析 | Markdown → 块列表 | 列表嵌套、代码块、段落边界 |
| 块 UI | 每块独立输入 | 光标跨块、合并/拆分段落 |
| 拖拽交互 | 手柄、浮层、插入指示线 | 与滚动、键盘、格式栏手势冲突 |
| 写回 Markdown | 重排后拼接 | 顺序变化不能破坏语法 |
| 撤销/重做 | 块级操作栈 | 现有整篇快照 undo 需重写 |
| 格式工具栏 | 绑定「当前聚焦块」 | 比单 `TextFieldValue` 复杂 |

存储仍可保持 Markdown（GitHub 同步无需改协议），但 **编辑态必须引入 Block 中间层**。

### 成本分级（仅「内容拖拽」）

#### A. 完整 Notion 体验（多类型块 + 真拖拽）

- **工期**：数周～约 2 个月+（单人业余可能更长）
- **范围**：重写 [`EditorScreen`](../app/src/main/java/com/andriod/reader/ui/editor/EditorScreen.kt) / [`EditorViewModel`](../app/src/main/java/com/andriod/reader/ui/editor/EditorViewModel.kt)
- **与 CHANGELOG 关系**：同类的「富文本所见即所得」大工程

#### B. 简化版：空行分隔的「段落块」

- **规则**：按 `\n\n` 切段，每段一个输入框 + 左侧拖手柄
- **工期**：约 2～3 周
- **局限**：列表项难单独拖、标题/代码块等精细类型支持弱
- **定位**：弱化版 Notion，适合验证是否值得继续投入

#### C. 不做真拖拽：上移 / 下移

- **交互**：长按块 →「上移」「下移」，或块菜单调整顺序
- **工期**：约 3～5 天（仍需块模型，约为 B 的一半工作量）
- **定位**：有 reorder 能力，但无 Notion 拖动手感

#### D. 第三方编辑器（WebView / 商业 SDK）

- **工期**：视集成而定，可能快于自研 UI
- **风险**：Markdown 优先工作流可能被改变、包体增大、定制与 GitHub 同步策略需额外评估
- **结论**：对本项目 **不一定合适**

### 与现有功能的关联

| 功能 | 影响 |
|------|------|
| 键盘 + 格式工具栏 | 块编辑器需重新处理焦点、滚动、IME（参见 [editor-keyboard-toolbar-layout.md](editor-keyboard-toolbar-layout.md)） |
| TTS 朗读 | 只要保存结果仍是 Markdown，朗读链路可不变 |
| 单元测试 | [`MarkdownEditorActions`](../app/src/main/java/com/andriod/reader/ui/editor/MarkdownEditorActions.kt) 等需随块模型调整 |
| GitHub 同步 | 文件格式不变则影响小；块编辑只改变「如何产生 Markdown」 |

### 结论与建议

| 目标 | 建议 |
|------|------|
| 必须 Notion 式拖块 | 列为 **二期大功能**；优先做 **B（段落块）** 验证价值 |
| 偶尔调整段落顺序 | **C（上移/下移）** 性价比更高 |
| 短期不动编辑器架构 | 继续剪切/粘贴，或先做列表级能力 |

**推荐路径（若做）**：B（段落块 + 上移/下移）→ 用得顺手再升级为真拖拽（A）。

**一句话**：笔记内容拖拽成本高，因为当前是「整篇 Markdown 文本框」而非「块编辑器」；Notion 的拖拽是架构能力，不是手势补丁。

## 2. 附：列表「拖笔记进文件夹」（对比）

若需求是 **侧栏/列表里** 把笔记拖进文件夹（非正文块拖拽）：

| 项 | 评估 |
|----|------|
| 成本 | **中等**（约 3～5 天） |
| 数据层 | 需 `moveNoteToFolder`、更新 `sync-state.json` 路径、GitHub 旧路径删除/新路径上传 |
| UI | 长按拖动 + 文件夹 drop 高亮；与现有侧滑编辑/删除需分区（拖动手柄或长按模式） |
| 性价比 | 相对正文块拖拽 **更高**，可作为独立二期小功能 |

与第 1 节 **不要混为一谈**。
